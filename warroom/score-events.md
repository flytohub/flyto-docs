# Score Events -- Grade Change Tracking

Score events track grade transitions over time, providing an audit trail of how and why your security posture changed.

## Overview

Every time a score is recomputed, the system compares consecutive snapshots. When a grade boundary is crossed (e.g., A to B or C to B), a score event is recorded with the reasons for the change.

```
2025-02-12  A --> B  (82 --> 72)
  Reasons:
  - Code Security dropped 15 pts (new critical CVEs in 'backend' repo)
  - Grade capped: critical CVE detected

2025-03-01  B --> A  (72 --> 78)
  Reasons:
  - Code Security improved 20 pts (CVEs patched)
  - Grade cap lifted: no critical CVEs remaining
```

## API Endpoints

### Org-Level Score Events

```
GET /api/v1/code/orgs/{org_id}/score-events?days=90
```

Returns grade change events for the entire organization. The `days` parameter controls the lookback window (default: 90).

**Response:**

```json
{
  "events": [
    {
      "timestamp": "2025-02-12T08:30:00Z",
      "previous_grade": "A",
      "new_grade": "B",
      "previous_raw": 82,
      "new_raw": 72,
      "previous_display": 780,
      "new_display": 720,
      "reasons": [
        "Code Security dropped 15 pts",
        "Grade capped: critical CVE detected"
      ]
    }
  ]
}
```

### Domain-Level Score Events

```
GET /api/v1/code/pentests/{pentest_id}/score-events?days=90
```

Returns grade change events for a specific CTEM domain/pentest project.

## How Events Are Detected

Events are generated by **consecutive snapshot comparison**:

1. On every score recomputation (real-time or background), the system stores a snapshot of the raw score, grade, and per-category breakdown.
2. The new snapshot is compared against the previous snapshot.
3. If the grade changed, a score event is created.
4. Category-level deltas are analyzed to generate human-readable reasons.

Snapshots are stored indefinitely for audit purposes.

## Reason Generation

The system generates reasons by analyzing what changed between consecutive snapshots:

### Category Delta Detection

When a category score changes by more than **10 points**, it is flagged as a reason:

- "Code Security dropped 15 pts" -- triggered when the code security sub-score decreased by 15+
- "Attack Surface improved 12 pts" -- triggered when external posture improved significantly
- "Diligence dropped 11 pts" -- triggered by scan coverage decrease or MTTR increase

### Grade Cap Triggers

Grade cap activation or deactivation is always explicitly annotated:

- "Grade capped: critical CVE detected"
- "Grade capped: scan coverage below 50%"
- "Grade capped: MTTR exceeds 14 days"
- "Grade cap lifted: no critical CVEs remaining"
- "Grade cap lifted: scan coverage restored above 50%"

### Cross-Dimensional Changes

Changes from [cross-dimensional modifiers](/warroom/scoring-methodology) are also noted:

- "Blast radius modifier changed: -3 to -7"
- "MTTR penalty increased: -4 to -8"
- "Pentest verdict: clean (+3)"

## Score Smoothing (30-Day Window)

Scores do not change instantaneously. A 30-day rolling window limits how fast the score can move:

| Time Since Last Score | Maximum Change |
|----------------------|----------------|
| 1 day | +/-0.5 pts |
| 7 days | +/-3.5 pts |
| 15 days | +/-7.5 pts |
| 30 days | +/-15 pts |

**Smoothing rate:** approximately 0.5 points per day, linear.

**What is NOT smoothed:**

- Grade caps (applied after smoothing, take effect immediately)
- Per-finding risk scores (only the overall org score is smoothed)

This means a critical CVE can cause an immediate grade drop from A to B even if smoothing would have kept the raw score above 75, because the grade cap overrides smoothing.

See [Scoring Methodology](/warroom/scoring-methodology) for full smoothing details.

## Grade Caps (3 Conditions)

Three hard conditions cap the maximum achievable grade at B (raw score <= 75):

| Condition | Cap |
|-----------|-----|
| Unpatched critical CVE in any repository | Max B |
| Scan coverage below 50% of repositories | Max B |
| Org-wide MTTR exceeds 14 days | Max B |

Grade caps are evaluated **after** smoothing and **always** take priority. They are the most common source of sudden grade drops and are always annotated in score event reasons.

## Using Score Events

Score events are useful for:

- **Executive reporting** -- Show grade trends and reasons over time
- **Incident investigation** -- Trace why a grade dropped on a specific date
- **Compliance** -- Demonstrate continuous monitoring and remediation progress
- **Alerting** -- Trigger notifications on grade downgrades via [webhooks](/warroom/api#webhooks-5-event-types)
